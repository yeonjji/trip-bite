## ADR-001: Supabase를 API 캐시로 사용

### 상태
**채택됨** (2026-03-11)

### 배경
TourAPI, 식약처 레시피 API 등 공공 API는 일일 호출 제한이 있고 응답 속도가 느린 편이다. 사용자에게 빠른 응답을 제공하면서도 API 호출 제한을 준수해야 한다.

### 선택지
1. **직접 API 호출**: 매 요청마다 외부 API 호출
2. **Redis 캐시**: Redis에 응답 캐시
3. **Supabase (PostgreSQL) 캐시**: Supabase DB에 정형화된 캐시 저장

### 결정
**선택지 3: Supabase를 API 캐시로 사용**

### 근거
- 이미 Supabase를 Backend로 사용하므로 추가 인프라 불필요
- 정형화된 스키마로 저장하면 다국어 데이터 보강(title_en, overview_en 등)이 용이
- SQL 쿼리로 복잡한 필터링/정렬/검색이 가능
- PostGIS로 위치 기반 검색 지원
- pg_cron으로 배치 갱신 자동화
- Redis 대비 운영 복잡성 감소 (무료 플랜 활용 가능)

### 트레이드오프
- Redis 대비 읽기 속도가 약간 느림 (그러나 10ms 이내로 충분)
- 스토리지 사용량 증가 (Supabase 무료 500MB 제한 주의)
- 캐시 무효화 로직을 직접 구현해야 함

## ADR-002: Server Components 기본 사용

### 상태
**채택됨** (2026-03-11)

### 배경
Next.js 15 App Router는 React Server Components(RSC)를 기본으로 사용한다. 클라이언트 컴포넌트와 서버 컴포넌트의 경계를 명확히 해야 한다.

### 선택지
1. **대부분 Server Components**: 데이터 페칭은 서버에서, 인터랙션만 클라이언트
2. **대부분 Client Components**: SPA 스타일, 클라이언트에서 모든 처리
3. **하이브리드 (상황별 결정)**: 명확한 규칙 없이 상황별 선택

### 결정
**선택지 1: Server Components 기본, 필요한 경우만 Client Components**

### 서버/클라이언트 구분 기준

| Server Components (기본) | Client Components ('use client') |
|-------------------------|--------------------------------|
| 페이지 컴포넌트 (page.tsx) | 필터 컴포넌트 (상태 관리) |
| 레이아웃 (layout.tsx) | 검색바 (입력 처리) |
| 카드 컴포넌트 (정적 표시) | 지도 컴포넌트 (Naver Maps API) |
| SEO 메타데이터 | 이미지 갤러리 (인터랙션) |
| 데이터 페칭 로직 | 캐러셀 (스와이프) |
| 접근성 배지 | 언어 전환 버튼 |
| | 리뷰 작성 폼 |
| | 모바일 네비게이션 (토글) |

### 근거
- SEO: 서버에서 렌더링하므로 검색 엔진 크롤링에 유리
- 번들 크기: 클라이언트 JS 최소화
- Supabase 직접 접근: 서버에서 service_role 키로 안전하게 DB 조회
- 성능: TTFB 개선, 스트리밍 SSR 활용 가능

### 트레이드오프
- 서버/클라이언트 경계에서 직렬화 가능한 데이터만 전달 가능
- 클라이언트 상태 관리가 필요한 부분은 별도 클라이언트 컴포넌트로 분리 필요

## ADR-003: next-intl을 i18n 라이브러리로 채택

### 상태
**채택됨** (2026-03-11)

### 배경
한국어와 영어 두 가지 언어를 지원해야 한다. App Router와 Server Components를 지원하는 i18n 라이브러리가 필요하다.

### 선택지
1. **next-intl**: App Router 네이티브 지원, Server Components 호환
2. **next-i18next**: Pages Router 중심, App Router 지원 제한적
3. **react-intl (FormatJS)**: 범용적이지만 Next.js App Router 설정 복잡
4. **자체 구현**: JSON 파일 로드 + 커스텀 훅

### 결정
**선택지 1: next-intl**

### 근거
- App Router + Server Components 완벽 지원
- `[locale]` 기반 라우팅 네이티브 지원
- `localePrefix: 'as-needed'` 옵션으로 기본 로케일(ko) prefix 생략 가능
- 가벼운 번들 크기
- 활발한 유지보수, 좋은 문서

### 설정 요약
- 로케일: `ko` (기본), `en`
- 라우팅: `[locale]` 세그먼트, `as-needed` prefix
- 번역 파일: `messages/ko.json`, `messages/en.json`
- 미들웨어: next-intl 미들웨어로 로케일 감지/리다이렉트

## ADR-004: 기본 로케일 prefix 없음 (as-needed)

### 상태
**채택됨** (2026-03-11)

### 배경
한국어가 기본 로케일이고 주 사용자가 한국인이므로, URL에 불필요한 `/ko` prefix를 넣지 않는 것이 자연스럽다.

### 선택지
1. **always**: 모든 로케일에 prefix (`/ko/travel`, `/en/travel`)
2. **as-needed**: 기본 로케일만 prefix 없음 (`/travel`, `/en/travel`)
3. **never**: 모든 로케일에 prefix 없음 (쿠키/헤더로 감지)

### 결정
**선택지 2: as-needed**

### URL 패턴

| 경로 | 한국어 (기본) | 영어 |
|------|-------------|------|
| 홈 | `/` | `/en` |
| 여행지 | `/travel` | `/en/travel` |
| 여행지 상세 | `/travel/abc123` | `/en/travel/abc123` |
| 맛집 | `/restaurants` | `/en/restaurants` |
| 검색 | `/search?q=경복궁` | `/en/search?q=gyeongbokgung` |

### 근거
- 한국어 사용자에게 깔끔한 URL 제공
- 영어 사용자에게는 `/en` prefix로 명확한 로케일 표시
- SEO: hreflang 태그로 검색 엔진에 다국어 페이지 연결
- next-intl의 `as-needed` 옵션으로 쉽게 구현

## ADR-005: shadcn/ui 커스텀 테마 사용

### 상태
**채택됨** (2026-03-11)

### 배경
UI 컴포넌트를 빠르게 구현하면서도 디자인 일관성과 접근성을 보장해야 한다.

### 선택지
1. **shadcn/ui**: Radix UI 기반, 커스터마이징 가능, 접근성 내장
2. **MUI (Material UI)**: Google Material Design, 무거운 번들
3. **Chakra UI**: 좋은 DX, 번들 크기 큼
4. **직접 구현**: Tailwind만 사용

### 결정
**선택지 1: shadcn/ui + warm theme 커스터마이징**

### 근거
- Radix UI 기반 접근성 (ARIA, 키보드 네비게이션) 기본 제공
- 컴포넌트 소스를 직접 소유 → 완전한 커스터마이징
- Tailwind CSS와 자연스러운 통합
- 개별 컴포넌트 import → 번들 최적화
- 빠른 프로토타이핑 가능

### 커스터마이징 범위
- CSS 변수를 warm 컬러로 오버라이드 (orange 계열)
- 카드 border-radius 16px
- 버튼 border-radius 12px
- 폰트: Pretendard

## ADR-006: SVG 한국 지도 사용

### 상태
**채택됨** (2026-03-11)

### 배경
사용자가 지역을 직관적으로 선택할 수 있는 UI가 필요하다. 한국 지도를 표시하고 지역을 클릭하면 해당 지역 허브로 이동하는 기능이다.

### 선택지
1. **SVG 한국 지도**: 커스텀 SVG path로 17개 시도 표현
2. **네이버/카카오 지도**: 실제 지도 위에 지역 오버레이
3. **이미지 맵**: PNG/JPG + HTML image map
4. **Canvas 기반**: D3.js + TopoJSON

### 결정
**선택지 1: SVG 한국 지도**

### 근거
- **가벼움**: SVG 파일 하나로 지도 완성 (수 KB)
- **커스터마이징**: CSS로 호버 효과, 선택 상태, 색상 변경 가능
- **반응형**: 벡터이므로 모든 해상도에서 선명
- **접근성**: 각 path에 `role`, `aria-label` 추가 가능
- **성능**: 외부 스크립트 로드 불필요
- **인터랙션**: React 이벤트 핸들러 직접 바인딩 가능

### 트레이드오프
- 실제 지도 수준의 상세한 지리 정보 제공 불가
- SVG path 데이터를 별도로 구해야 함
- 시군구 수준의 세부 지역 표현은 복잡

## ADR-007: 네이버 지도 API를 주 지도 서비스로 사용

### 상태
**채택됨** (2026-03-11)

### 배경
여행지/맛집 상세 페이지에 위치를 지도에 표시해야 한다. 국내 사용자가 주 타겟이다.

### 선택지
1. **네이버 지도 API**: 국내 상세 데이터, 높은 친숙도
2. **카카오 지도 API**: 국내 상세 데이터, 카카오 생태계 연동
3. **Google Maps**: 글로벌 표준, 외국인 친숙
4. **Mapbox**: 커스터마이징 자유도 높음
5. **네이버 주 + 카카오 보조**: 네이버 기본, 카카오 폴백/길찾기

### 결정
**선택지 5: 네이버 지도 주 사용 + 카카오 지도 보조**

### 역할 분담

| 기능 | 네이버 지도 | 카카오 지도 |
|------|-----------|-----------|
| 상세 페이지 위치 표시 | O (주) | X |
| 지역 허브 전체 마커 | O (주) | X |
| 길찾기 연동 | O (네이버 지도 앱) | O (카카오맵 앱) |
| 장소 검색 보조 | X | O (로컬 API) |
| 장애 시 폴백 | - | O |

### 근거
- 국내 사용자 70%+ 네이버 지도 친숙
- 네이버 지도 JavaScript API v3: 안정적, 풍부한 기능
- 카카오 로컬 API: 주변 맛집/카페 추천에 유용
- 카카오맵 길찾기 링크: 모바일에서 카카오맵 앱 연동 편리
- Google Maps 대비 국내 데이터 품질 우수

### 트레이드오프
- 외국인 사용자에게는 Google Maps가 더 친숙할 수 있음
- 두 지도 서비스 API 키 관리 필요
- 네이버 지도 스크립트 + 카카오 지도 스크립트 동시 로드 시 번들 증가 → 동적 import로 해결

## ADR-008: 기상청 API + 3시간 캐시

### 상태
**채택됨** (2026-03-11)

### 배경
여행지 상세 페이지와 지역 허브에 날씨 정보를 제공해야 한다. 날씨 데이터는 실시간성이 중요하지만, API 호출 비용과 외부 API 의존도를 관리해야 한다.

### 선택지
1. **기상청 API 직접 호출**: 매 요청마다 기상청 API 호출
2. **기상청 API + 1시간 캐시**: 짧은 캐시로 높은 실시간성
3. **기상청 API + 3시간 캐시**: 기상청 발표 주기에 맞춘 캐시
4. **서드파티 날씨 API (OpenWeather 등)**: 글로벌 서비스, 유료

### 결정
**선택지 3: 기상청 API + 3시간 캐시 (weather_cache 테이블, 지역 대표 날씨)**

### 근거
- 기상청 단기예보는 3시간 간격으로 발표 (0200, 0500, 0800, ...)
- 발표 주기에 맞춰 캐시하면 불필요한 API 호출 최소화
- 공공 데이터이므로 무료 사용 가능
- 국내 날씨 데이터 정확도 최고 (기상청 공식)
- `weather_cache` 테이블에 정형화 저장 → 계절별 추천 로직에 활용 가능
- MVP에서는 `area_code`당 대표 격자 좌표 1개만 유지하여 운영 복잡도를 낮출 수 있음

### 캐시 전략
```
1. 사용자 요청 → weather_cache 조회
2. cached_at이 3시간 이내 → 캐시 반환
3. 3시간 초과 → 기상청 API 호출 → weather_cache 갱신 → 반환
4. pg_cron: 3시간마다 17개 시도 날씨 배치 갱신
5. 7일 이상 오래된 캐시 자동 삭제
```

### 트레이드오프
- 최대 3시간 지연된 날씨 정보 (여행 정보 용도로는 충분)
- 개별 장소 초정밀 예보가 아니라 지역 대표 날씨이므로 광역 지역에서는 오차가 있을 수 있음
- 기상청 API 응답 형식이 복잡하여 파싱 로직 필요 (카테고리 코드 → 사람이 읽을 수 있는 형태)
- 격자좌표 변환 매핑 테이블 필요

## ADR-009: 자체 리뷰 시스템 우선 (MVP)

### 상태
**채택됨** (2026-03-11)

### 배경
여행지/맛집/캠핑장에 대한 리뷰와 평점이 필요하다. 외부 서비스(네이버 플레이스, 카카오맵) 링크를 제공할 수는 있지만, 외부 리뷰 자체를 수집·적재하는 것은 MVP 단계에서 구현/운영 리스크가 크다.

### 선택지
1. **자체 리뷰 시스템만**: Supabase Auth 기반 리뷰/평점
2. **외부 장소 링크만**: 네이버/카카오 상세 페이지 링크 제공
3. **자체 리뷰 + 외부 링크 병행**: 내부 리뷰 집계, 외부 상세 링크 보조
4. **리뷰 없이 출시**: 리뷰 기능 제외

### 결정
**선택지 3: 자체 리뷰 + 외부 링크 병행**

### Phase별 전략

**Phase 1 (MVP):**
- Supabase Auth 기반 사용자 인증 (이메일/비밀번호)
- 인증 사용자만 리뷰/평점 작성 가능
- `reviews` 테이블은 사용자 작성 리뷰만 저장
- RLS 정책: 본인 리뷰만 수정/삭제
- 트리거로 `destinations.avg_rating`, `camping_sites.avg_rating` 자동 갱신

**Phase 2 (확장):**
- 네이버 검색 API / 카카오 로컬 API로 장소 매칭
- 장소 상세 페이지에서 네이버 지도 / 카카오맵 이동 링크 제공
- 외부 리뷰 원문은 서비스 DB에 적재하지 않음
- 내부 평점 집계는 자체 리뷰만 사용

### 근거
- MVP에서는 핵심 기능(탐색/검색)에 집중, 리뷰는 기본만
- Supabase Auth → reviews RLS가 간단하게 구현 가능
- 외부 리뷰 크롤링은 법적 이슈, API 제한, 데이터 품질 등 리스크가 큼
- 공식 장소 검색 API가 제공하는 범위는 장소 검색/상세 링크 수준이므로, 링크 제공까지를 공식 지원 범위로 보는 편이 안전함
- 자체 리뷰 데이터가 축적된 후 외부 상세 링크를 함께 제공하면 UX 보강이 가능

### 트레이드오프
- MVP 초기에는 리뷰 수가 적어 신뢰도 낮을 수 있음
- 사용자 가입 필요 (탐색만 하는 사용자에겐 허들)
- 외부 링크에 의존하므로 서비스 내부에서 외부 리뷰 요약/집계를 제공하지 못함

## ADR-010: 캠핑장 데이터를 별도 테이블(camping_sites)로 분리

### 상태
**채택됨** (2026-03-11)

### 배경
고캠핑 API의 캠핑장 데이터를 저장할 구조를 결정해야 한다. 캠핑장은 기존 destinations(여행지/맛집)와 동일한 "장소" 개념이지만, 데이터 구조가 상당히 다르다. 고캠핑 API는 사이트 유형(일반/자동차/글램핑/카라반), 바닥 타입(잔디/파쇄석/데크/자갈/맨흙 각각의 사이트 수), 운영 기간(계절별), 부대시설, 화로대 사용 여부 등 캠핑 전용 필드가 15개 이상이다.

### 선택지
1. **destinations 테이블 확장**: 기존 destinations에 캠핑 전용 컬럼을 추가하거나, `raw_data` JSONB에 모든 캠핑 데이터를 저장
2. **별도 테이블 (camping_sites)**: 캠핑장 전용 테이블을 신규 생성

### 결정
**선택지 2: 별도 테이블 (camping_sites)**

### 근거
- **필드 차이**: 캠핑 전용 필드가 15개 이상 (사이트 유형별 수, 바닥 타입별 수, 운영 기간, 부대시설, 반려동물, 트레일러/카라반 동반 등). destinations에 추가하면 대부분의 여행지/맛집 행에서 NULL 컬럼이 대량 발생
- **타입 안전성**: JSONB에 모든 캠핑 데이터를 넣으면 TypeScript 타입 안전성이 저하되고, 필터링 쿼리가 복잡해짐
- **인덱싱 최적화**: 캠핑 전용 필터(업종, 바닥 타입, 반려동물, 시설)에 대한 인덱스를 효율적으로 설계 가능
- **데이터 소스 분리**: destinations는 TourAPI, camping_sites는 고캠핑 API에서 각각 독립적으로 캐시/갱신
- **쿼리 성능**: 캠핑장 목록 조회 시 destinations의 수만 건 데이터와 분리되어 빠른 조회 가능
- **확장성**: 캠핑장에 특화된 기능(사이트 예약 연동 등)을 추가할 때 독립적으로 확장 가능

### 트레이드오프
- 통합 검색 시 destinations와 camping_sites 두 테이블을 모두 조회해야 함 (UNION 또는 병렬 쿼리)
- reviews 테이블에 `camping_site_id` FK를 추가하고 집계 트리거도 함께 확장해야 함
- 지역 허브 페이지에서 양쪽 데이터를 병합하여 표시해야 함
- 코드에서 "장소" 관련 공통 로직이 분산될 수 있음

### 대안 검토
- destinations 확장 방식은 초기에는 간단하지만, 캠핑 필터링 쿼리가 복잡해지고 `raw_data` JSONB 의존도가 높아져 유지보수가 어려워짐
- 캠핑 전용 필드가 5개 이하였다면 destinations 확장이 더 나았겠지만, 15개 이상의 전용 필드는 별도 테이블이 적합
